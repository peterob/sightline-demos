🔧 Key System Components Explained

🔄Projection Maintenance Triggers

❌ The Problem

Our system uses Event Sourcing - all changes are stored as immutable events in the audit log. But users need to query current state quickly (like "What's our current budget balance?").

Reading events every time would be too slow for real-time dashboards.

✅ The Solution

Projection tables store the current state, automatically updated from events. Think of them as "materialized views" that stay in sync.

Users get fast queries while maintaining cryptographic audit integrity.

📋 Example: $3,500 Payment to Green Thumb Landscaping
1
Event Created: PaymentExecuted event added to audit_events table
INSERT INTO audit_events (event_type, event_data, amount, vendor_id) VALUES ('PaymentExecuted', '{"amount": 3500, "vendor": "Green Thumb"}', ...)
2
Trigger Fires: Database automatically detects the new event
3
Projections Updated: Multiple read tables updated instantly
-- Update payment tracking UPDATE payment_tracking SET status = 'completed' WHERE id = payment_id; -- Update budget burn rate UPDATE budget_burn_projection SET spent_amount = spent_amount + 3500 WHERE category = 'landscaping'; -- Update vendor status UPDATE vendor_status_projection SET last_payment_date = NOW() WHERE vendor_id = 'green_thumb_id';
4
Users See Updates: Dashboard shows new budget balance immediately
🛠️ Technical Implementation
-- PostgreSQL trigger function CREATE OR REPLACE FUNCTION update_projections() RETURNS TRIGGER AS $$ BEGIN -- Parse the event and update relevant projections CASE NEW.event_type WHEN 'PaymentExecuted' THEN -- Update payment_tracking table -- Update budget_burn_projection table -- Update vendor_status_projection table WHEN 'VendorApproved' THEN -- Update vendor_status_projection table WHEN 'BudgetAllocated' THEN -- Update budget_burn_projection table END CASE; RETURN NEW; END; $$ LANGUAGE plpgsql; -- Attach trigger to audit_events table CREATE TRIGGER maintain_projections AFTER INSERT ON audit_events FOR EACH ROW EXECUTE FUNCTION update_projections();

⚙️Template Configuration Engine

❌ The Problem

Each HOA thinks their approval process is unique, but 95% are actually very similar. If we let them customize everything, we'd have:

  • Complex custom code for each client
  • Performance problems
  • Maintenance nightmares
  • Security vulnerabilities

✅ The Solution

Pre-built templates that handle 95% of use cases. HOAs pick a template and configure simple parameters.

The engine automatically generates policies, approval workflows, and spending limits.

💰Budget Tracking and Validation

❌ The Problem

HOAs need to prevent overspending, but traditional systems have gaps:

  • Budget checks happen too late
  • Multiple people can approve simultaneously
  • No real-time tracking
  • Easy to manipulate budget numbers

✅ The Solution

Real-time cryptographic budget validation that happens before approval, with mathematical proof of availability.

Every approval includes cryptographic proof that funds were available at that moment.

💳 Example: $4,200 HVAC Repair Request
1
Request Submitted: Tom submits emergency HVAC repair
Current Maintenance Budget: • Allocated: $35,000 • Committed: $22,100 (approved but not paid) • Spent: $8,750 (actually paid) • Available: $4,150 ⚠️
2
Budget Validation: System checks availability cryptographically
BUDGET VALIDATION FAILED Requested: $4,200 Available: $4,150 Shortfall: $50 Status: REJECTED - Insufficient funds Recommendation: Request budget transfer or reduce scope
3
Alternative Action: Sarah (President) approves $500 budget transfer
4
Re-validation: Now sufficient funds available, approval proceeds
5
Cryptographic Proof: Approval includes mathematical proof of budget availability
🔢 Budget Math & Validation
-- Real-time budget calculation function CREATE OR REPLACE FUNCTION check_budget_availability( p_hoa_id UUID, p_category VARCHAR(50), p_amount DECIMAL(10,2) ) RETURNS TABLE( available BOOLEAN, current_available DECIMAL(10,2), budget_proof_hash VARCHAR(64) ) AS $$ DECLARE budget_record RECORD; proof_data TEXT; BEGIN -- Get current budget state SELECT * INTO budget_record FROM budget_burn_projection WHERE hoa_id = p_hoa_id AND category = p_category; -- Calculate availability IF budget_record.available_amount >= p_amount THEN -- Generate cryptographic proof of availability proof_data := budget_record.allocated_amount::text || budget_record.committed_amount::text || budget_record.spent_amount::text || p_amount::text || NOW()::text; RETURN QUERY SELECT true, budget_record.available_amount, encode(digest(proof_data, 'sha256'), 'hex'); ELSE RETURN QUERY SELECT false, budget_record.available_amount, NULL::VARCHAR(64); END IF; END; $$ LANGUAGE plpgsql;

📱Notification System

❌ The Problem

HOA approval workflows involve multiple people across different organizations:

  • Volunteer board members check email sporadically
  • External accountants need timely notifications
  • Property managers work different schedules
  • Approvals get stuck waiting for responses

✅ The Solution

Smart notification system that routes messages based on urgency, role, and preferences.

Integrates with the cryptographic audit trail to provide context and proof.

🎯 Why These Components Matter

Together, these four components transform the user experience:

  • Projection Triggers: Make the system feel instant and responsive
  • Template Engine: Make onboarding effortless (no custom configuration needed)
  • Budget Validation: Make overspending mathematically impossible
  • Notifications: Make approvals happen quickly across organizations

Without these, you'd have a technically sound but practically unusable system. With them, you have a system that feels magical while being cryptographically secure.
