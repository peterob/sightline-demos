🗄️ Append-Only Log Approach

1
Procurement team submits vendor "Acme Corp"
→ Creates log entry with entity creation
Log Entry: {action: "create", entity: "vendor_456", new_value: {name: "Acme Corp", status: "pending_approval"}}
✓ Single append operation to log table
2
Finance manager requests documentation
→ Creates field change entry + note entry
Log Entry 1: {action: "field_change", field: "status", old: "pending_approval", new: "documentation_required"}
Log Entry 2: {action: "add_note", field: "notes", new_value: "Need W-9 tax form"}
✓ Two separate append operations
3
Vendor provides W-9 form
→ Creates file upload entry
Log Entry: {action: "field_change", field: "tax_document", old: null, new: "w9_form_uploaded.pdf"}
✓ File stored + log entry appended
4
Finance manager approves vendor
→ Creates status change entry
Log Entry: {action: "field_change", field: "status", old: "documentation_required", new: "approved"}
✓ Status updated in log
5
Department head creates $5,000 PO
→ Creates new PO entity
Log Entry: {action: "create", entity: "po_789", new_value: {vendor_id: "vendor_456", amount: 5000, status: "pending_approval"}}
✓ New entity chain started in log
6
Department head approves PO
→ Updates PO status
Log Entry: {action: "field_change", entity: "po_789", field: "status", old: "pending_approval", new: "approved"}
✓ PO ready for payment processing
7
AP clerk initiates payment
→ Creates payment entity + triggers external system
Log Entry: {action: "create", entity: "payment_101", new_value: {po_id: "po_789", amount: 5000, status: "initiated"}}
✓ Payment request sent to bank API
8
System receives payment confirmation
→ Updates payment status
Log Entry: {action: "field_change", entity: "payment_101", field: "status", old: "initiated", new: "completed"}
✓ Workflow complete

Data Access Pattern:

To get current vendor status: Query all log entries for vendor_456, replay chronologically to build current state

To find pending approvals: Query all entities, replay their logs, filter by current status

Storage: Linear growth, simple structure, single table

⚡ Event Sourcing Approach

1
Procurement team submits vendor "Acme Corp"
→ Publishes VendorSubmittedForApproval event
Event: VendorSubmittedForApproval {vendor_name: "Acme Corp", contact_email: "john@acme.com", service_category: "IT Services"}
✓ Event stored + Vendor projection updated + Approval queue projection updated
2
Finance manager requests documentation
→ Publishes DocumentationRequested event
Event: DocumentationRequested {required_documents: ["W-9"], reason: "Tax documentation required", due_date: "2025-05-31"}
✓ Event stored + Vendor projection updated + Task queue projection updated + Email notification sent
3
Vendor provides W-9 form
→ Publishes DocumentUploaded event
Event: DocumentUploaded {document_type: "W-9", file_path: "docs/vendor_456/w9_form.pdf", uploaded_by: "procurement_001"}
✓ Event stored + Document projection updated + Approval queue projection updated
4
Finance manager approves vendor
→ Publishes VendorApproved event
Event: VendorApproved {approved_by: "finance_manager_002", approval_notes: "All documentation complete", effective_date: "2025-05-24"}
✓ Event stored + Vendor projection updated + Approved vendor list updated + Notification sent
5
Department head creates $5,000 PO
→ Publishes PurchaseOrderCreated event
Event: PurchaseOrderCreated {vendor_id: "vendor_456", amount: 5000, description: "Software development services", created_by: "dept_head_003"}
✓ Event stored + PO projection created + Approval queue projection updated
6
Department head approves PO
→ Publishes PurchaseOrderApproved event
Event: PurchaseOrderApproved {approved_by: "dept_head_003", approval_amount: 5000}
✓ Event stored + PO projection updated + Payment queue projection updated
7
AP clerk initiates payment
→ Publishes PaymentInitiated event
Event: PaymentInitiated {po_id: "po_789", vendor_id: "vendor_456", amount: 5000, payment_method: "ACH", expected_date: "2025-05-26"}
✓ Event stored + Payment projection created + Bank API called + Tracking projection updated
8
System receives payment confirmation
→ Publishes PaymentCompleted event
Event: PaymentCompleted {transaction_id: "txn_abc123", actual_amount: 5000, completion_date: "2025-05-26"}
✓ Event stored + Payment projection updated + Accounting system notified + Audit trail updated + Vendor notified

Data Access Pattern:

To get current vendor status: Query Vendor projection directly (pre-computed)

To find pending approvals: Query Approval Queue projection directly

Storage: Event store + multiple optimized projections for different query patterns

🔍 Key Architectural Differences

Append-Only Log

  • Single write path: All changes go to one log table
  • Read complexity: Must replay log to get current state
  • Query performance: Slower for complex queries
  • Integration: Manual polling/watching of log
  • Semantics: Field-level changes, less business context

Event Sourcing

  • Multiple projections: Optimized read models for different needs
  • Read simplicity: Direct queries on projections
  • Query performance: Fast, purpose-built views
  • Integration: Natural event-driven architecture
  • Semantics: Rich business events with full context
