1. Product scope

Core product statement: before a covered AP payment is released, the vendor confirms the amount, invoice, payment date, and destination-account last four.

Core invariant: a covered payment cannot become release_ready until the vendor has confirmed the payment details for that specific Work Order.

Build now

  • Buyer dashboard for high-risk AP payment Work Orders.
  • Vendor and vendor contact records.
  • Secure Trade Message link.
  • Vendor actions: confirm, request change, dispute.
  • Immutable event log.
  • Work Order state transitions.
  • Proof record after confirmation.
  • Basic confirmation metrics.

Do not build yet

  • PO confirmation.
  • Seller-side AR postings or mirrored ledgers.
  • NetSuite or ERP export.
  • Modern Treasury payment initiation.
  • Generated journal entries.
  • Full postings table.
  • Full cases workflow.
  • GPU, grants, intercompany, or trade-finance verticals.
  • Public verifier replay or formal verification.

4. Core data model

The prototype data spine uses events, vendors, work_orders, and policy_versions, plus operational helper tables for organizations, users, contacts, confirmation links, and delivery attempts.

organizations

CREATE TABLE organizations (
  organization_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

users

CREATE TABLE users (
  user_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL REFERENCES organizations(organization_id),
  email TEXT NOT NULL,
  full_name TEXT,
  role TEXT NOT NULL DEFAULT 'finance_user',
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (organization_id, email)
);

vendors

CREATE TABLE vendors (
  vendor_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL REFERENCES organizations(organization_id),
  legal_name TEXT NOT NULL,
  vendor_external_ref TEXT,
  trust_tier TEXT NOT NULL DEFAULT 'provisional',
  bank_last4 TEXT,
  bank_verified_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (organization_id, legal_name)
);

vendor_contacts

CREATE TABLE vendor_contacts (
  vendor_contact_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  vendor_id UUID NOT NULL REFERENCES vendors(vendor_id),
  full_name TEXT,
  email TEXT,
  phone TEXT,
  email_verified_at TIMESTAMPTZ,
  phone_verified_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CHECK (email IS NOT NULL OR phone IS NOT NULL)
);

policy_versions

CREATE TABLE policy_versions (
  policy_version_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL REFERENCES organizations(organization_id),
  version_label TEXT NOT NULL,
  trigger_rules JSONB NOT NULL DEFAULT '{}'::jsonb,
  auto_approve_rules JSONB NOT NULL DEFAULT '{}'::jsonb,
  activated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  superseded_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX one_active_policy_per_org
ON policy_versions (organization_id)
WHERE superseded_at IS NULL;

Core tables continued

work_orders

CREATE TABLE work_orders (
  work_order_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL REFERENCES organizations(organization_id),
  vendor_id UUID NOT NULL REFERENCES vendors(vendor_id),
  vendor_contact_id UUID REFERENCES vendor_contacts(vendor_contact_id),
  policy_version_id UUID NOT NULL REFERENCES policy_versions(policy_version_id),
  cast_reference TEXT NOT NULL UNIQUE,
  invoice_number TEXT NOT NULL,
  invoice_date DATE,
  amount NUMERIC(18,2) NOT NULL,
  currency TEXT NOT NULL DEFAULT 'USD',
  payment_date DATE,
  bank_last4 TEXT,
  trigger_reason TEXT NOT NULL,
  source_event_id UUID,
  bilateral_state TEXT NOT NULL DEFAULT 'proposed',
  control_state TEXT NOT NULL DEFAULT 'pending',
  release_state TEXT NOT NULL DEFAULT 'held',
  work_order_hash TEXT,
  sent_at TIMESTAMPTZ,
  viewed_at TIMESTAMPTZ,
  vendor_acted_at TIMESTAMPTZ,
  expires_at TIMESTAMPTZ,
  created_by UUID REFERENCES users(user_id),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CHECK (bilateral_state IN ('proposed','sent','viewed','confirmed','change_requested','disputed','expired')),
  CHECK (control_state IN ('pending','approved','blocked')),
  CHECK (release_state IN ('held','release_ready','blocked','expired'))
);

events

CREATE TABLE events (
  event_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL REFERENCES organizations(organization_id),
  work_order_id UUID REFERENCES work_orders(work_order_id),
  policy_version_id UUID REFERENCES policy_versions(policy_version_id),
  event_type TEXT NOT NULL,
  actor_type TEXT NOT NULL,
  actor_id TEXT,
  actor_display_name TEXT,
  payload JSONB NOT NULL DEFAULT '{}'::jsonb,
  prior_event_id UUID REFERENCES events(event_id),
  lineage_hash TEXT NOT NULL,
  occurred_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CHECK (actor_type IN ('human','vendor','system','agent'))
);

CREATE RULE no_update_events AS ON UPDATE TO events DO INSTEAD NOTHING;
CREATE RULE no_delete_events AS ON DELETE TO events DO INSTEAD NOTHING;

CREATE TABLE confirmation_links (
  confirmation_link_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

6. State machine

FlowState pathOutcome
Happy pathproposed → sent → viewed → confirmed → approved → release_readyPayment can proceed.
Change requestedproposed → sent → viewed → change_requested → blockedPayment remains held.
Disputeproposed → sent → viewed → disputed → blockedPayment remains held.
No responseproposed → sent/viewed → expiredPayment expires and cannot proceed.
Release rule: release_state can become release_ready only when bilateral_state = confirmed and control_state = approved.

7. Work Order hash

The vendor confirmation should bind to the exact payment record. Hash the fields the vendor sees and the governing policy version.

type WorkOrderHashInput = {
  organization_id: string;
  work_order_id: string;
  vendor_id: string;
  vendor_contact_id: string;
  invoice_number: string;
  amount: string;
  currency: string;
  payment_date: string | null;
  bank_last4: string | null;
  policy_version_id: string;
};

function computeWorkOrderHash(input: WorkOrderHashInput): string {
  return sha256(canonicalize(input));
}

On confirmation, recompute the hash. If it differs from the stored hash, reject the confirmation and keep the payment held.

10. Main services

ServiceResponsibilities
Event serviceSingle way to append events and compute lineage hashes.
Work Order serviceCreate, send, view, confirm, request change, dispute, expire, evaluate policy.
Delivery serviceCreate confirmation links, send email/SMS, record delivery attempts.
Policy serviceApprove only confirmed Work Orders; block disputed/change-requested Work Orders.
 
function evaluatePolicy(workOrder: WorkOrder): PolicyResult {
  if (workOrder.bilateral_state !== "confirmed") {
    return { approved: false, reason: "Vendor confirmation required before release." };
  }
  return { approved: true, reason: "Vendor confirmed payment details." };
}

11. Security requirements

  • Generate 32-byte random confirmation tokens.
  • Store only SHA-256 token hashes.
  • Links expire in 24-48 hours.
  • Vendor action links are single-use.
  • Rate-limit token routes.
  • Store only bank-account last four.
  • Never store full bank-account numbers.
  • Buyer users can only access their organization records.
  • Vendor portal must not expose buyer dashboard data.
Start with secure link + optional OTP. Add passkeys after the confirmation loop is operational.

12. Jobs

JobLogic
ExpirationFind sent/viewed Work Orders past expires_at; append vendor.expired; set release state to expired.
Delivery retryRetry failed email/SMS delivery up to configured limit.
MetricsCompute sent, viewed, confirmed, change-requested, disputed, expired, and median confirmation time.

15. MVP definition of done

  1. Buyer can create a high-risk AP payment intent.
  2. Buyer can send a Trade Message.
  3. Vendor can open a secure link on mobile.
  4. Vendor can confirm, request change, or dispute.
  5. Confirmation creates an immutable event.
  6. Work Order becomes release_ready only after confirmation.
  7. Request change or dispute blocks release.
  8. Expiration prevents stale confirmation.
  9. Buyer can view proof record.
  10. Metrics dashboard shows confirmation performance.
