Bilateral Event

The canonical unit of truth. The only primitive that can anchor a Posting.

A BilateralEvent is a co-signed, immutable, hash-anchored record confirmed by both parties. It cannot be created unilaterally. It cannot be modified after creation. It is the architectural pivot — everything before it is uncontrolled, everything after flows deterministically.

Key constraints

Non-negotiables: Requires counterparty confirmation · Immutable after creation · Only source of Postings (FK enforced at schema level) · PolicyVersion pinned at creation time, permanently

State machine

ProposedEvent written
↓ Trade Message sent (SMS or email)
bilateral_state = 'sent'
↓ Vendor opens message
bilateral_state = 'viewed'
↓
┌ Vendor taps Confirm → bilateral_state = 'confirmed' → BilateralEvent written
├ Vendor taps Change → bilateral_state = 'disputed' → Case opened
└ SLA expires → bilateral_state = 'timeout' → Case opened

Lineage hash

-- SHA-256 computed at write time, verified at settlement
lineage_hash = SHA256(
  previous_event_hash  -- chains to prior event
  || event_type
  || actor_id
  || policy_version_id   -- pinned at write time
  || payload_json
)

Modify any historical event → break every hash forward. An auditor verifying the root hash knows the entire history is intact without reconstructing it.

See also: decisions/rejected → Unilateral event model for why this architecture was chosen over simpler alternatives.

Posting

The double-entry ledger entry. Can only exist if a BilateralEvent created it.

Schema constraint
bilateral_event_id NOT NULL
Enforced at the database level. Not application code. Not a rule. A structural impossibility.

The audit test

Pull any Posting. Find its bilateral_event_id. Look it up in events. Verify the lineage_hash matches. If the hash is broken, something was tampered with. If the FK is null, the Posting is invalid and should not be in the books.

What CAST does not own

CAST generates the Posting. The ERP stores the number. CAST stores the proof. If the ERP entry is modified and the lineage hash breaks, CAST flags it. CAST is not replacing the ERP — it is the proof layer that sits alongside it.

Case

The structured exception handler. What opens when policy cannot auto-resolve.

A Case is a typed, timestamped object created whenever any primitive hits a condition the policy engine cannot resolve. It is not an email thread. It carries the full context of what was found, why it was blocked, who is assigned, and what is needed to proceed.

When a Case opens

Triggerblocking_reason
Vendor taps Change on Trade MessageVENDOR_DISPUTED
Bilateral state times out past SLACONFIRMATION_TIMEOUT
Policy auto-approve fails conditionsPOLICY_BLOCKED
Budget would go negativeBUDGET_EXCEEDED
Counterparty on blocked listCOUNTERPARTY_BLOCKED
Lineage hash mismatch detectedHASH_TAMPER_DETECTED
Agent acts outside delegation scopeDELEGATION_EXCEEDED
The safe degradation pattern: An agent that hits a boundary does not guess and does not fail silently. It opens a Case with the full event trail, the specific blocking reason, what human input is needed, and the SLA deadline. The human receives structured context — not an unexplained error.

Rejected Approaches

What was considered, why it was rejected, and what we chose instead. The most valuable page in the wiki for avoiding re-litigated decisions.

Unilateral event model
Rejected — foundational

A unilateral event log is a better database — it proves nothing to a counterparty. CAST's value is shared truth creation. Without bilateral confirmation, the proof is not portable to a regulator, lender, or insurer.

Chose instead: Bilateral co-authorship as schema requirement. See bilateral-event →

OPA / Rego for policy engine (v1)
Rejected — April 2026

OPA adds a runtime dependency and a new DSL (Rego) the team doesn't own. v1 policy rules are known, finite, and stable. Debugging a Rego failure during the bilateral event loop is harder than TypeScript. TLA+ sits above the policy engine — OPA adds complexity without verifiability.

Chose instead: Deterministic TypeScript evaluation. Revisit in v2 if policy complexity grows.

CDC / Debezium for event streaming
Rejected — early 2026

Debezium captures DB-level row changes, not business events. The lineage hash cannot be computed at the DB layer. BilateralEvent is a business concept — CDC treats all row writes equally. Adds Kafka operational complexity before the loop is proven.

Chose instead: Application-level event writes with pgcrypto SHA-256.

Full ERP replacement
Rejected — foundational

ERP replacement is a 3–5 year enterprise sale with deep integration requirements before any customer can use it. CAST's value is the bilateral proof layer, not the GL. ERPs are commodity — proof is the differentiator.

Chose instead: CAST exports journal entries. ERPs store numbers. CAST stores proof.

Sell-side as first build target
Rejected — April 2026

The sell-side use case requires a new design partner (Macquarie), new event taxonomy, new policy definitions, and a new GTM — six months of unvalidated risk before the core product is proven.

Chose instead: CAST wiki first (Karpathy pattern on home turf), then pitch sell-side. See wiki-pattern →

Database Schema — v1

Four tables. This is the entire prototype schema. Steps 6–10 are locked until the bilateral loop is proven.

-- 1. events (append-only — NO UPDATE or DELETE ever)
CREATE TABLE events (
  id                UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  event_type        TEXT        NOT NULL,
  bilateral_state   TEXT        NOT NULL DEFAULT 'proposed',
  actor_id          TEXT        NOT NULL,
  actor_type        TEXT        NOT NULL,  -- 'human' | 'agent'
  delegation_id     UUID,                    -- required if actor_type = 'agent'
  policy_version_id UUID        NOT NULL,
  payload           JSONB       NOT NULL,
  lineage_hash      TEXT        NOT NULL,  -- SHA-256(prev_hash || payload)
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 2. work_orders
CREATE TABLE work_orders (
  id                UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  proposed_event_id UUID        NOT NULL REFERENCES events(id),
  status            TEXT        NOT NULL DEFAULT 'open',
  delivery_channel  TEXT,                    -- 'sms' | 'email'
  assignee_id       TEXT,
  sla_deadline      TIMESTAMPTZ,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 3. policy_versions
CREATE TABLE policy_versions (
  id              UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  version_tag     TEXT        NOT NULL UNIQUE,   -- e.g. 'POL-v4.2'
  rules           JSONB       NOT NULL,
  effective_from  TIMESTAMPTZ NOT NULL,
  authored_by     TEXT        NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 4. postings — bilateral_event_id is NEVER NULL
CREATE TABLE postings (
  id                 UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  bilateral_event_id UUID         NOT NULL REFERENCES events(id), ← NEVER NULL
  policy_version_id  UUID         NOT NULL REFERENCES policy_versions(id),
  debit_account      TEXT         NOT NULL,
  credit_account     TEXT         NOT NULL,
  amount             NUMERIC(18,2) NOT NULL,
  currency           TEXT         NOT NULL DEFAULT 'USD',
  lineage_hash       TEXT         NOT NULL,
  created_at         TIMESTAMPTZ  NOT NULL DEFAULT now()
);
Steps 6–10 are locked. GL assignment, budget checks, ERP export — do not build these until the bilateral event loop is proven with a real vendor confirming a real Trade Message.

Karpathy Wiki Pattern

How and why CAST uses the LLM wiki accumulation pattern for its own development.

The problem

Every Claude Code session previously started by rediscovering context from prior conversations. The spec lives in a Word document. Design decisions live in chat logs. Architecture debates are not queryable. This is the RAG problem — re-deriving from scratch on every session.

The three-file system

CLAUDE.md
Repo root. Auto-loaded every session. Primitives, schema, decisions, 5 questions to ask before writing code.
wiki/
Obsidian vault of linked Markdown pages. One page per concept. Viewable as a knowledge graph.
wiki-compiler-prompt.md
The idea file. Paste into Claude Code with the spec to build or update the vault. Karpathy's pattern: share the idea, let the agent build it for your context.

The accumulation loop

New session with new context
↓
Read CLAUDE.md + wiki/index.md
↓
Read new material (spec update, conversation, decision)
↓
UPDATE relevant wiki pages — no duplicates
↓
Run lint pass → lint-report.md
↓
Show diff of what changed
RAG retrieves and forgets. Wiki accumulates and compounds. The 10th Claude Code session is faster than the 1st. The Macquarie pitch reuses the same pattern — deal events build the wiki automatically, no manual ingest step.
