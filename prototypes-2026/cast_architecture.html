What this document is

The architectural reference for CAST's five-layer system. Each section contains a diagram, a one-paragraph description, and — where applicable — the build rule or exit criterion for that layer. This document is meant to sit open alongside your editor during coding.

What this document is not

This is not the build guide. For session-by-session prompts and exit criteria, see CAST_Layer0_Build_Guide.md. For the canonical spec, see CAST_System_Design_v1_3_March2026.docx.

Three rules that govern every layer

1. agent_id is non-nullable on every event — this is the architectural claim, not a feature.
2. Every ledger posting carries a foreign key to a bilateral event. No posting without a signed event.
3. No posting reaches the ledger until the bilateral confirmation is cryptographically verified.

Overview — The five layers

Each layer above Layer 0 is a read surface on the event ledger below it.

The ledger is the spine. Every layer above Layer 0 is a read surface — a projection, a query, a verification endpoint. Nothing bypasses the ledger. If the three non-negotiables hold at Layer 0, every upper layer inherits correctness. If any one fails, every upper layer inherits the defect.

Build rule: no upper layer ships until Layer 0 has produced ~10 real bilateral events.

Building now

Layer 0 — Bilateral event loop

The write path. Cryptographic confirmation gates every posting.

The foundation. A work order is issued internally, delivered to a counterparty over SMS, email, or API, and confirmed with a cryptographic signature (passkey). Only after that signature is verified does a ledger posting become legal. The bilateral event is the only thing that binds both parties to the same fact, and the posting's foreign key to that event is what makes the whole architecture hold.

Non-negotiables enforced here:
(1) agent_id non-nullable on every event   (2) posting FK → bilateral event   (3) no posting until signature verified

Exit criterion: one real counterparty completes the loop end-to-end, demoable via shareable URL.

After Layer 0 produces ~10 real events

Layer 1 — Auditor view

Read-only lineage projection. Any posting traces back to its event chain.

The first read surface. Given any ledger posting, Layer 1 reconstructs the complete chain of events that produced it — the originating work order, the delivery mechanism, the signature, the timestamp, the agent_id. No write access. It is a projection, not a system of record.

Gate: do not build until ~10 real bilateral events exist in the ledger. Building against synthetic data optimizes for imagined shape, not real shape.

Next

Layer 2 — Verification API

Third-party attestation. Yes/no answer, no lineage exposed.

The commercial surface for Scope B buyers. A third party — auditor, lender, platform owner — posts a signature to the verification endpoint and receives an attestation: valid or not, signed by whom, at what time. No lineage, no context, no access to the underlying system. Platform owners (Anthropic, Microsoft, Salesforce) can expose this to their enterprise customers so agent actions become auditable without exposing the system that produced them.

Later

Layer 3 — Vendor event layer

Vendor's view across all buyers. Portable proof, emergent trust tiers.

The vendor's own view of every bilateral event they've co-authored, across all their buyers. This is where network effects begin: each new buyer adds to every vendor's provable history. A vendor with 200 signed events is a different credit risk than one with 2. Vendors become portable across platforms without losing their reputation — their proof bundle travels with them.

v2+ — only viable after enough events exist to compile from

Layer 4 — Compiled control plane

The operator surface. Compiles event history into enforceable rules.

The operator surface, viable only after the system has accumulated enough real events to compile from. Invariants become enforceable rules. Proof requirements become policy. Decisions become deterministic. This is the v2 operating system — it is in the roadmap specifically so that nothing upstream is built toward it prematurely. Compiling rules before enough events exist produces brittle policy that will be thrown away.

Conditional layer: "if at all." Build only when event volume and operator count justify it.
