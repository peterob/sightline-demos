What this proves · 1
Both parties saw the same details.
The buyer wrote the payload. The seller signed sha256(payload). The signature is invalid against any other version of the payload. If a single character of amount, invoice, or destination differed, this signature would not verify.
What this proves · 2
The seller — not someone who controlled the seller's email — confirmed.
The signature was produced by a passkey enrolled to agt_cascade_b2. Owning the inbox does not produce a valid signature. A future verifier checks the signature against the public key on file at enrollment.
What this proves · 3
The record cannot drift.
The signed bytes are sha256(payload_version || canonical_json(payload)). The canonicalizer is committed code. Adding a column to events in 2027 does not invalidate this signature from 2026.
What this proves · 4
The architecture would have blocked a posting without this row.
postings.event_id is NOT NULL REFERENCES events(event_id). No ledger entry can exist without an event behind it. The constraint is in SQL, not in application code.
