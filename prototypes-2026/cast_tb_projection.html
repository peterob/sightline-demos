CAST · CORPORATE FINANCE LAYOUT Trial balance as projection — not artifact.
Schema · a16z Corporate Finance Function

The trial balance is a query, not a table.

Traditional ERPs store balances. Every post updates a balance row, every period locks it, every restatement rewrites it. CaST does not. The events table is the source of truth; postings are immutable contributions; the trial balance is what falls out when you ask the right question of them. This page shows the three layers, the four parameters, and the four views you get for free.

01 · The three layers

One event log. One postings table. Many projections.

Every fact the firm cares about — a management fee invoice, a payroll run, a contractor SOW, an FX rate publication, a period close — enters as a signed event. Each event generates one or more postings, foreign-keyed back to it. Trial balances, P&Ls, NAVs, and tax provisions are computed on demand from those postings.

Layer 1 · Source of truth Events — signed, immutable, append-only.

Each row carries a registered agent, a versioned policy reference, a canonical payload, and a hash. No row is ever updated or deleted.

02 · The four parameters

Every TB is the same query with four knobs turned.

Mgmt-co GAAP, fund NAV, Singapore statutory, tax-basis provision — all are the same projection function with different parameter values. No parallel ledgers. No reconciliations between them, because there is nothing to reconcile.

① Entity scope

One entity, a parent + subs, or the consolidation set. Each entity has its own functional currency, jurisdiction, and fiscal calendar. New entities are additive — sign a setup event, postings start flowing.

② Book

mgmt_co, fund, consol, tax, stat. A single event posts into multiple books simultaneously — the rules for which postings go where are themselves versioned policies.

③ Period

Jurisdiction-scoped. US mgmt co on calendar months; Singapore sub on its own fiscal calendar. A period close is a signed event, not a stored flag — it anchors which events are final for the window.

④ Presentation currency

Translation uses signed FX rate events. P&L at average rate, balance sheet at period-end, equity at historical. CTA falls out as the plug — computed, never manually entered.

03 · Four views, one event log

04 · What this replaces

Stored balances vs. projected balances.

The architectural choice is not "better reporting." It is which artifact is the source of truth. Once balances stop being data, half the operational machinery a traditional ERP requires simply has nothing to do.

Traditional ERP

The TB is a stored table.

  • gl_balances row updated on every journal post
  • Period-end procedure to roll balances forward
  • Reconciliation reports proving balances tie to journals
  • FX revaluation runs that rewrite balance rows
  • Restatement = rewriting historical balance rows
  • Parallel ledgers for tax, statutory, fund accounting
  • "TB as of date X" requires a stored snapshot
  • Audit support assembled from multiple extracts

CaST

The TB is a query.

  • Postings are immutable, no balance table to maintain
  • Period-close is a signed event, not a procedure
  • Nothing to reconcile — TB is the postings, projected
  • FX revaluation is a query against signed FX events
  • Restatement = new policy version, replay projection
  • One event log; books are projection parameters
  • "TB as of date X" is just the projection at that timestamp
  • Audit support is one join from any TB line
