Why this matters

The product cannot win if first-run vendor onboarding feels like enterprise software. V1 trust should feel closer to secure checkout than vendor master maintenance. Later versions can add persistent signer roles, hardware-backed credentials, and portal-only modes for higher-risk counterparties.

Request change

Dispute / fraud block

Shared workflow state

V1 should track three parallel state machines: bilateral response, internal control, and downstream settlement / posting. This keeps trade truth separate from accounting execution.

Core buyer-side features

  • Invoice intake via email, upload, ERP sync, or API
  • Trigger engine for first payment, bank change, threshold, duplicate risk, and rush requests
  • Message composer with structured payment-intent preview
  • AP queue with bilateral state, control state, and escalation tools
  • Approval workbench with policy checks, signer validation, and anomaly review
  • ERP / AP export with CAST reference and proof metadata
  • Settlement tracking and posting-ready status
  • Vendor trust administration for channels, signers, and bank-token rules

Core vendor-side features

  • Mobile-first inbound message experience
  • Lightweight authentication: magic link, OTP, then passkey upgrade for repeat use
  • One-tap confirm flow
  • Structured request-change flow for machine-readable corrections
  • Dispute and fraud-block flow that immediately halts payment
  • Message history for repeat counterparties
  • Visible trust cues: buyer identity, past relationship, and approved payment destination indicators

Shared system functionality

  • Three-track state model: bilateral, control, settlement/posting
  • Immutable event log for all transitions
  • Notification and reminder engine
  • Role-based access for AP, controller, approver, and vendor signer
  • Audit trail with event timestamps, actor, reason code, and attached evidence
  • API layer for ERP/AP integrations
  • Reporting for adoption, confirmation latency, disputes, auto-approval rate, and covered spend

MVP build order

  • 1. Structured trade message object and state model
  • 2. Buyer queue + message composer + vendor message page
  • 3. Confirm / change / dispute actions
  • 4. Basic control rules and approval workbench
  • 5. ERP export and settlement sync
  • 6. Notifications, reminders, and reporting
  • 7. Better identity and signer controls for repeat usage
