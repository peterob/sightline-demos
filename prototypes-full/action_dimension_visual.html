The Semantic Corruption in Traditional Accounting

Why treating Actions as Accounts destroys causal information

Problem Traditional 3D Framework
All treated as "Accounts" (same dimension)
Cash
Inventory
Payables
↕️ mixed with ↕️
Expenses
Revenue

"Expenses" and "Revenue" are recorded as accounts,
debited and credited just like Cash or Inventory

🔴 What's Lost

When you record "Debit: Expenses, Credit: Cash" you lose what action caused this. Was it a Purchase? A Payment? A Fee? The cause is collapsed into a generic "Expense" account, making variance analysis and causal attribution impossible to automate.

Solution 4D Frame of Reference
Dimension: ACTION (Causes)
Expenditure
Expense
Revenue
Payment
Receipt
Dimension: ACCOUNTS (Effects)
Cash
Inventory
Payables
Receivables
Earnings

🟢 What's Preserved

Actions and Accounts are separate dimensions. An action (like "Purchase") causes flows between accounts (Payables → Inventory). The cause-effect relationship is explicit, enabling automatic explanation of any balance change.

The Same Economic Event, Two Representations

Scenario: Company purchases $500 of raw materials on credit

Traditional Double-Entry
// Two entries, no explicit cause
Debit: Inventory $500
Credit: Payables $500

// ❌ What ACTION caused this?
// ❌ Was it a Purchase? Transfer? Adjustment?
// ❌ Information destroyed at recording time
4D Binary Transaction
// Single 4-tuple, cause preserved
{
  flow: $500,
  action: "Expenditure:Purchase",
  tail: "Payables",
  head: "Inventory"
}

// ✓ Cause (Purchase) explicitly recorded
// ✓ Enables automatic variance attribution

Why This Matters for Sightline

Capability Traditional (3D) Sightline (4D)

Audit Trail Journal entries are semantically ambiguous. Same debit/credit pattern can mean different things. Every event has explicit action type. Impossible to record semantically invalid events.
