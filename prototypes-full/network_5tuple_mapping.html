Network 5-Tuple → Financial 5-Tuple

Borrowing battle-tested architecture from 40 years of network engineering

The Deep Structural Parallel

Network engineers solved the problem of uniquely identifying flows, tracking state, and enforcing policy across distributed systems decades ago. The 5-tuple is the foundation of firewalls, NAT, load balancers, and flow analytics. Sightline's financial 5-tuple can inherit this entire architectural playbook.

🔄 Element-by-Element Mapping
🌐 Network 5-Tuple
1 Source IP Address Origin endpoint
2 Destination IP Address Target endpoint
3 Source Port Origin process
4 Destination Port Target service
5 Protocol Transport method
💰 Financial 5-Tuple
1 Tail (Source Account) Origin of value
2 Head (Destination Account) Target of value
3 Policy Authorization context
4 Action Transaction type
5 Flow Value amount

💡 The Mapping Insight

Source/Destination IP → Tail/Head Accounts: Both identify endpoints of a flow.

Port Numbers → Policy + Action: Ports identify the service/process context. Policy + Action identify the business context (what authorization, what type of transaction).

Protocol → (Implicit in Action hierarchy): TCP/UDP/ICMP define transport semantics. Physical/Financial action types define transaction semantics.

📦 Architectural Patterns to Borrow

Stateful Flow Tables High Value

Network: Firewalls maintain connection state tables indexed by 5-tuple. Return traffic auto-permitted for established sessions.
Financial application: Maintain "transaction state tables" indexed by (Policy, Action, Direction). When a PO is created, pre-authorize the matching invoice and payment flows. The policy establishes the session; subsequent events are validated against session state.
// Policy PO-2024-0847 creates "session"
// Auto-permits: Invoice matching this PO
// Auto-permits: Payment matching this invoice
// Rejects: Payment without prior invoice

Flow Export (NetFlow/IPFIX) High Value

Network: Export summarized flow records instead of raw packets. Enables analytics at scale without storing every packet.
Financial application: Export summarized "financial flow records" for analytics. Don't query raw events for dashboards—export aggregated flows by (Policy, Action, Direction) with counters and timestamps.
FlowRecord {
  policy: "budget-2024-capex",
  action: "expenditure",
  direction: (payables, assets),
  total_flow: $2.4M,
  event_count: 847,
  first_seen: "2024-01-15",
  last_seen: "2024-12-10"
}

NAT / Address Translation High Value

Network: NAT translates between internal and external addresses while maintaining session continuity via 5-tuple mapping.
Financial application: "Policy Translation" for multi-party coordination. Each party has internal policies; a translation layer maps to shared external representation. Internal: "Dept-Budget-Q4" → External: "PO-2024-0847".
// Buyer's internal view
policy: "capex-gpu-cluster"

// Shared/external view (translation)
policy: "PO-2024-0847"

// Seller's internal view
policy: "customer-acme-order-4521"

Connection Timeout / Aging Medium Value

Network: Flow table entries expire after inactivity. Prevents stale state accumulation. Different timeouts for TCP vs UDP.
Financial application: Policies have validity periods and inactivity timeouts. A PO unused for 90 days triggers review. Different "protocols" (expenditure vs payment) have different timeout semantics.
Policy {
  valid_until: "2024-12-31",
  inactivity_timeout: "90d",
  on_timeout: "flag_for_review"
}

Load Balancing / Hash Distribution Medium Value

Network: Hash 5-tuple to distribute traffic across servers while maintaining session affinity.
Financial application: Hash (Policy, Action, Direction) to distribute event processing across workers while ensuring all events for a policy are processed by the same node (for state consistency).
worker_id = hash(policy_id) % num_workers
// All events for PO-2024-0847 go to worker 3
// Ensures policy state is consistent

Deep Packet Inspection → Deep Event Inspection Medium Value

Network: Look beyond headers into payload for application identification and threat detection.
Financial application: Look beyond the 5-tuple into event payload (line items, attachments, metadata) for anomaly detection, compliance checks, and fraud analysis.
// 5-tuple says: valid expenditure
// Deep inspection finds:
// - Line item "consulting" on hardware PO
// - Vendor address changed recently
// → Flag for review

SIEM / Flow Correlation High Value

Network: Correlate flows across devices and time to detect attack patterns and track threat actors.
Financial application: Correlate events across policies and time to detect anomalies. "This vendor received payments from 3 different budgets in 30 days" or "This expense pattern matches known procurement fraud."
Alert: Vendor ACME-CORP
- Payment from budget-sales: $50K
- Payment from budget-eng: $47K
- Payment from budget-ops: $52K
Pattern: Split to avoid $50K approval threshold?

QoS / Priority Queuing Low Value Now

Network: Classify traffic by 5-tuple for priority treatment and bandwidth allocation.
Financial application: Priority processing for certain policy types. Board-approved CapEx processes before routine OpEx. Compliance-critical events process before analytics queries.
queue_priority = {
  "board_approved": 1,
  "compliance_required": 2,
  "standard": 3,
  "analytics": 4
}
🔥 The Stateful Firewall Parallel

🌐 Network Stateful Firewall

1
Outbound SYN packet arrives

5-tuple extracted, policy checked against rules

2
State table entry created

5-tuple → {state: SYN_SENT, timeout: 30s}

3
Return SYN-ACK arrives

Reverse 5-tuple matches state entry → permitted

4
Connection established

State updated, subsequent packets auto-permitted

5
FIN or timeout

State entry removed, flow complete

💰 Financial Stateful Policy Engine

1
Purchase Order created

5-tuple extracted, validated against budget policy

2
Policy session created

5-tuple → {state: PO_ISSUED, capacity: $100K}

3
Invoice arrives

Matches PO policy session → auto-validated

4
Payment authorized

State updated, capacity decremented

5
PO fulfilled or expired

Policy session closed, audit trail complete

🎯 The Strategic Insight

Network 5-tuple infrastructure has been battle-tested for 40 years. Every router, firewall, and load balancer implements these patterns. By mapping Sightline's financial 5-tuple to the same architectural primitives, you inherit:


1. Proven scalability patterns — Flow tables handle millions of concurrent sessions
2. Established mental models — Network engineers already understand stateful flow tracking
3. Mature tooling patterns — NetFlow collectors, SIEM integration, flow analytics
4. Clear operational semantics — Timeouts, state machines, policy enforcement
5. Interoperability patterns — Standard export formats, collector architectures


This is not metaphor—it's architecture. The same data structures, the same state machines, the same export protocols. Financial events flow through Sightline like packets flow through a network.
