Traditional ERPs store balances. CaST does not. Events are the source of truth, postings are immutable contributions, the trial balance is what falls out when you ask the right question of them.

What the controller sees is a familiar trial balance — accounts, debits, credits, totals tying to zero. The format is unchanged because that's what the auditor expects.

What the schema guarantees is that every line on the TB is a deterministic function of signed events. There is no balance table to maintain, no period roll-forward, no parallel ledger to reconcile. The TB at any point in time, in any currency, for any entity scope, is the same query with four parameters turned.

Three scenes follow. Each one is a layer of the stack. The fourth is what you no longer need.

01 Layer · Source of Truth

Events — signed, immutable, append-only.

Every fact the firm cares about enters as a signed event. Each row carries a registered agent, a versioned policy reference, a canonical payload, and a hash. No row is ever updated. No row is ever deleted. The auditor in 2031 verifies these against the same canonicalizer that signed them in 2026.

03 Layer · Computed View

The Trial Balance — a function of the postings, not a row.

Set the four knobs. The TB returns. Same inputs, same output, every time, forever. No stored balances. No period roll-forward. No reconciliation between the TB and the ledger — they are the same thing, asked differently.

The projection function · committed code · v1

// pure function · same inputs always return the same TB · replayable in 2031 function trial_balance(entity_scope, book, period, present_ccy) →   SELECT account, SUM(translate(amount, currency, present_ccy, fx_event_id))   FROM postings   WHERE entity_id IN entity_scope     AND book = book     AND period_id IN periods_in(period)     AND event_id IN (SELECT event_id FROM events WHERE verified)   GROUP BY account
04 Layer · Same Event, Four Views

One event log — mgmt-co, fund, tax, consol.

The Q1 management fee event produces postings in four books simultaneously. Fund accounting, mgmt-co accounting, tax, and consolidation are not parallel teams reconciling to each other. They are four queries over one substrate.

05 What this replaces

Stored balances vs. projected balances.

The architectural choice is not "better reporting." It is which artifact is the source of truth. Once balances stop being data and become an answer to a question about data, half the operational machinery a traditional ERP requires has nothing to do.

Traditional ERP

The TB is a stored table.

  • gl_balances row updated on every journal post
  • Period-end procedure rolls balances forward
  • Reconciliation reports prove balances tie to journals
  • FX revaluation runs rewrite balance rows in place
  • Restatement = rewriting historical balance rows
  • Parallel ledgers for tax, statutory, fund accounting
  • "TB as of date X" requires a stored snapshot
  • Audit support assembled from multiple extracts

CaST

The TB is a query.

  • Postings are immutable — no balance table to maintain
  • Period close is a signed event, not a procedure
  • Nothing to reconcile — TB is the postings, projected
  • FX revaluation is a query against signed FX events
  • Restatement = new policy version + replay projection
  • One event log; books are projection parameters
  • "TB as of date X" is the projection at that timestamp
  • Audit support is one join from any TB line
