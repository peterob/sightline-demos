Proof-Based Settlement Architecture

Risks, assumptions, and the path to deterministic verification

1. Assumption Inventory

Every "proof" relies on underlying assumptions. Here's what observation-based settlement visibility assumes:

Assumption Risk If Wrong Can Be Eliminated?
Bank/Plaid data is accurate Your "proof" is based on false data No — must trust data source
Bank/Plaid data is complete Missing transactions = invisible events No — can't prove negative
Your instruction caused the observed settlement Correlation ≠ causation Yes — with unique identifiers
No tampering between source and your system Man-in-the-middle modifies data Yes — with signed payloads
Your system's clock is accurate Timestamp manipulation Yes — trusted timestamping
Your event store is immutable Historical records modified Yes — hash chains / Merkle trees
Settlement finality is real ACH returns, chargebacks reverse "settled" transactions Partially — must model reversibility

2. Risk Analysis

⚠️ Risk 1: Correlation vs. Causation

You observe: "We instructed payment X, and payment X settled." But did YOUR instruction cause it?

Scenario: Client's accountant manually initiates the same payment in their bank portal 5 minutes before your system does. Both "succeed." You see settlement, claim credit. But your instruction was actually rejected as duplicate. Your audit trail shows false causation.
✓ Mitigation: Idempotency Keys

Include a unique identifier (UUID) in every payment instruction. Require that identifier to appear in the bank's confirmation. If the bank doesn't echo your key, you can't claim causation.

⚠️ Risk 2: Data Source Integrity

You're trusting Plaid/bank APIs to give you accurate, complete data. What if they're wrong?

Scenario: Plaid has a bug. It shows a transaction as "posted" when it's actually "pending." You mark settlement complete. Next day, the transaction reverses. Your proof was based on false data.
✓ Mitigation: Multi-Source Corroboration

Don't trust a single source. Corroborate Plaid transaction with bank statement import AND accounting system entry. Three-way match for settlement, not just PO/Invoice/Receipt.

⚠️ Risk 3: Observation Latency

There's always a gap between when something happens and when you observe it. State is uncertain during that window.

Scenario: Payment settles at 2:00 PM. Plaid webhook arrives at 2:15 PM. At 2:10 PM, someone queries your system for settlement status. You say "pending." They make a business decision based on stale data.
✓ Mitigation: Explicit Uncertainty Modeling

Never claim certainty you don't have. Model states as: instructed → initiated → pending_confirmation → confirmed. Report confidence intervals, not binary states.

⚠️ Risk 4: Settlement Non-Finality

ACH isn't final for 60 days. Chargebacks can reverse card payments. "Settled" doesn't mean "irreversible."

Scenario: Payment settles. You release goods. 45 days later, ACH return (NSF). You thought settlement was final. It wasn't. Your proof of settlement is now a proof of reversal.
✓ Mitigation: Finality Tiers

Model settlement finality explicitly: provisional (0-3 days), likely_final (3-60 days for ACH), final (past chargeback window). Never treat provisional as final.

⚠️ Risk 5: Completeness (Proving the Negative)

You can prove what you observed. You cannot prove you observed everything.

Scenario: Client has two bank accounts. You're integrated with one. Fraudulent payments go out of the second account. Your audit trail shows "no unauthorized payments" because you didn't see them.
✓ Mitigation: Scope Declaration

Explicitly declare the scope of your visibility. "This proof covers accounts A, B, C for period X-Y." Never imply completeness beyond declared scope.

3. What CAN Be Proof-Based & Deterministic

✓ Proof 1: You Received This Data At This Time

Cryptographically provable with signed timestamps.

// RFC 3161 Trusted Timestamping { "data": { "transaction_id": "TXN-123", "amount": 47250.00, "status": "settled" }, "data_hash": "SHA256(data) = a3f8c2d1...", "timestamp": "2026-02-06T14:23:01Z", "tsa_signature": "RFC3161_TSA_signs(data_hash + timestamp)", "tsa_certificate": "DigiCert Timestamp Authority" } // Verification: Anyone can verify TSA signature proves data existed at timestamp

What this proves: "This exact data payload existed at this exact time, signed by a trusted third party."

✓ Proof 2: Data Was Not Modified After Receipt

Hash chains provide tamper-evidence.

// Each event includes hash of previous event Event N: { data: {...}, prev_hash: "hash(N-1)", hash: "hash(N)" } Event N+1: { data: {...}, prev_hash: "hash(N)", hash: "hash(N+1)" } // To tamper with Event N, attacker must: // 1. Modify Event N // 2. Recompute hash(N) // 3. Update Event N+1's prev_hash // 4. Recompute hash(N+1) // 5. Continue for ALL subsequent events // 6. Do this before anyone has verified the chain // Mitigation: Periodic anchoring to public blockchain or RFC 3161 TSA

What this proves: "If any event was modified, the chain breaks. An intact chain proves no tampering."

✓ Proof 3: Your Instruction Matches The Settlement (Causation Link)

Idempotency keys create verifiable linkage.

// Your instruction includes unique key Instruction: { idempotency_key: "SL-2026-0847-a3f8", vendor: "Pacific Steel", amount: 47250.00, account: "****6789" } // Bank API echoes key in confirmation Bank Response: { status: "initiated", reference: "ACH-9982341", client_reference: "SL-2026-0847-a3f8" // ← Your key echoed back } // Settlement confirmation (via Plaid or bank) Settlement: { transaction_id: "TXN-ABC123", amount: 47250.00, memo: "SL-2026-0847-a3f8" // ← Your key in memo field } // Verification: instruction.idempotency_key == settlement.memo // This PROVES your instruction caused this settlement

What this proves: "This specific settlement was caused by this specific instruction because the unique key appears in both."

✓ Proof 4: Deterministic State Reconstruction

Given the same event stream, anyone reaches the same state.

// State is a pure function of events function computeState(events: Event[]): State { return events.reduce((state, event) => { switch(event.type) { case 'payment.instructed': return { ...state, pending: [...state.pending, event.payload] }; case 'payment.settled': return { ...state, pending: state.pending.filter(p => p.id !== event.payload.id), settled: [...state.settled, event.payload] }; // ... deterministic for all event types } }, initialState); } // Proof: computeState(events) always returns same result for same events // Anyone can replay and verify

What this proves: "The state I show you is the only possible state given these events. You can verify by replaying."

4. The Deterministic Settlement Proof Architecture

🎯 Design Principle: Prove What You Can, Declare What You Assume

Don't claim proofs you can't deliver. Be explicit about trust boundaries.

Trust Boundary Model

Within Your Trust Boundary (Provable)
  • Data received at timestamp (RFC 3161)
  • Data not modified since receipt (hash chain)
  • Instruction-settlement linkage (idempotency key)
  • State computation determinism (pure functions)
  • Event ordering (causal DAG)
Outside Your Trust Boundary (Assumed)
  • Bank/Plaid data accuracy
  • Bank/Plaid data completeness
  • Settlement finality (until chargeback window closes)
  • No out-of-band transactions
  • Client's other systems don't contradict

6. The Honest Answer

What You Can Achieve

Deterministic, proof-based verification that:

  • You received specific data at specific times (RFC 3161 proves)
  • That data has not been modified (hash chain proves)
  • Your instruction and the settlement are linked (idempotency key proves)
  • Anyone can replay and reach the same state (determinism proves)

What You Cannot Achieve (Irreducible Assumptions)

No matter what you build, you must trust:

  • The data source (Plaid, bank) is giving you accurate data
  • You're seeing all relevant transactions (no hidden accounts)
  • Settlement finality rules of the payment rail (ACH returns, chargebacks)

This is why your Trust Architecture prototype scores data sources. You're making assumptions explicit and probabilistic, not eliminating them.

7. Recommendation: Layered Proof Model

Layer What It Proves Trust Requirement
Layer 0: Cryptographic Data integrity, timestamps, linkage Math only — trustless
Layer 1: Source Trust Data source is reliable Trust Plaid/bank — scored
Layer 2: Corroboration Multiple sources agree Reduces single-source risk
Layer 3: Finality Settlement won't reverse Time-based (60 days ACH)

Your proof certificates should declare which layers are satisfied. A "Layer 0 + Layer 1" proof is weaker than "Layer 0 + Layer 1 + Layer 2 + Layer 3".
