3. Event Taxonomy

Standardize all incoming events into these categories:

Event Type Category Sources Trust Level
payment.instructed Instruction Your system (policy engine triggered) Internal — Canonical
payment.approved Instruction Your system (human/agent approved) Internal — Canonical
payment.initiated Execution Bank API, Bill.com, Stripe High — Regulated processor
payment.pending Execution Bank API (ACH pending) High — Bank confirmation
payment.settled Confirmation Bank webhook, Plaid transaction High — Final settlement
payment.failed Exception Bank API, processor webhook High — Requires investigation
payment.returned Exception Bank (NSF, account closed) High — Reversal required
deposit.confirmed Confirmation Bank API, Plaid balance change High — Funds available

4. The Reconciliation Engine

Your policy engine should run continuous reconciliation:

Reconciliation Rules

// Rule 1: Instruction → Execution matching WHEN payment.instructed EXISTS AND payment.initiated NOT EXISTS WITHIN 24 hours THEN flag "Payment instruction not executed" // Rule 2: Execution → Settlement matching WHEN payment.initiated EXISTS AND payment.settled NOT EXISTS WITHIN 5 business days THEN flag "ACH settlement delayed" // Rule 3: Amount matching WHEN payment.instructed.amount != payment.settled.amount THEN flag "Settlement amount mismatch" // Rule 4: Orphan detection WHEN payment.settled EXISTS AND payment.instructed NOT EXISTS THEN flag "Unmatched settlement - potential fraud" // Rule 5: Reversal detection WHEN payment.settled EXISTS AND payment.returned EXISTS referencing same transaction THEN flag "Payment reversal - update AR/AP"

6. Minimum Viable Visibility Stack

🎯 MVP Integration Set (8-10 weeks)

With just three integrations, you achieve 80% visibility:

  1. Plaid — Settlement confirmation for any US bank account
  2. QuickBooks Online — Instruction capture (when user creates bill payment)
  3. Stripe or Shopify — E-commerce payment chain (pick based on target vertical)

This covers: Cash in bank ↔ AR/AP in accounting ↔ E-commerce transactions

8. What This Enables

Capability Without Handling Funds Audit Value
Prove payment was instructed ✅ Hash of instruction in your system "Our system generated this payment instruction at this time"
Prove payment was executed ✅ Bank/processor API confirmation "The bank confirms they initiated ACH transfer #X"
Prove payment settled ✅ Plaid transaction or bank webhook "Funds arrived in vendor's account on this date"
Detect missing settlements ✅ Reconciliation engine flags orphans "These 3 payments were instructed but never settled"
Replay historical state ✅ Event DAG allows point-in-time reconstruction "Here's exactly what our system knew on March 15"
Cryptographic proof of chain ✅ Hash linking proves no tampering "This chain is mathematically unmodified since creation"

⚠️ What You Can't Do Without Handling Funds

  • Guarantee execution — You can instruct, but client's bank ultimately decides
  • Conditional release (escrow) — Can't hold funds until conditions met
  • Real-time settlement — ACH is still T+1 to T+3
  • Payment failure prevention — Can only detect after the fact

These limitations are acceptable for your stated goal (audit visibility). If escrow becomes critical, revisit facilitation via BaaS partner.
